# v1.5 Release Roadmap

The work to add object-visibility clips to the timeline. One new conceptual primitive on top of a stable v1.

**v1.5 scope:** drop a 3D object from the OM onto a V-track; the clip's in/out points drive the object's render visibility. Object clips look visually distinct from camera clips (color + icon). Tracks are per-type — a track is either a camera track or an object track based on its first clip, not intermixed.

**Why this is its own release** (not folded into v1 or v2):
- Introduces a new conceptual primitive (non-camera clip type) — too much new surface to land inside v1's polish-and-ship window.
- Narrowly scoped — one behavior, no new evaluators or composition math. Different in kind from v2's motion-layers feature, which is a major surface.
- Calling it out as its own release makes the **one-off framing explicit**: this is the *only* non-camera/non-audio behavior the timeline will get. The release boundary is also a scoping fence.

**Depends on:** v1 shipping. See [v1-release-roadmap.md](v1-release-roadmap.md).

---

## Scoping principles (locked in up-front)

These are commitments, not preferences. They keep v1.5 from drifting into a generalized "Shotblocks for anything" rewrite.

1. **Object visibility is the only non-camera/non-audio behavior the timeline will support.** Not the start of "Shotblocks for materials / lights / transforms." If the question "why not materials too?" comes up post-ship, the answer is "by design — see v1.5 roadmap." This commitment goes in `.agent/constitution.md` when v1.5 is implemented.
2. **No new conceptual primitives beyond the new clip type.** Same drag/drop, same edit gestures, same persistence model as camera/audio clips. The clip just drives a different parameter on commit.
3. **No new UI panels or windows.** The Inspector grows a new section if needed; no separate Object panel.
4. **Reuse as much existing infrastructure as possible.** Render-resolver branches on existing `sourceType` field. Track header reuses existing layout. Settings dropdown reuses Inspector's dropdown.

## Plan sequence

| 1 | **Object-clip foundation** | concept | _not yet written_ | New `sourceType: 'object'`; OM-drag of an object creates an object clip; clip persists; per-track type (camera vs object) enforced |

Three plans. v1.5 ships when all three land verified.

---

## Order rationale

**Plan 1 (foundation) first** because the data model and per-track type enforcement need to land before any visual or evaluator work. Drag-from-OM → object clip on V-track, persists across save/load. No visibility change yet — the clip exists but doesn't do anything to the object.

**Plan 2 (evaluator) second** because once clips persist, the per-frame "what visibility should this object have?" question becomes meaningful. Visibility resolver runs in the same C++ path that routes cameras — same `EVMSG_TIMECHANGED` / playback-tick cadence.

**Plan 3 (visuals + inspector + conflict handling) last** because the visual distinction needs a working feature to design against, and the take-over-and-warn flow needs the conflict-detection plumbing from Plan 1.

---

## What v1.5 is NOT

- **No material / layer / transform timelining.** Visibility only. By design.
- **No keyframe animation on object clips.** Visibility is binary (on during clip, off outside). No fade-in/out, no opacity animation. If you want a fade, animate the object's opacity in C4D's own timeline; Shotblocks doesn't replace that.
- **No object-clip preview thumbnails.** Clips show their object's name + icon, not a render preview.
- **No intermixed tracks.** A V-track is either a camera track or an object track, not both. First clip dropped sets the type; subsequent drops of the wrong type are rejected (with a friendly explanation) or redirected to a different track.
- **No motion-layers-style sub-lanes for object clips.** v2 sub-lanes are camera-only.
- **No multi-object clips.** One clip = one object. Group of objects = one clip per object (or wrap them in a null and drop the null).

## Cross-plan decisions already locked in

(So they don't drift as plans get drafted.)

- **Track type:** per-track. First clip dropped sets the type. Mismatched drops rejected or redirected.
- **Visibility property driven:** render visibility (`MODE_ON` / `MODE_OFF`). Affects both viewport and final render so the user sees what's actually rendered while scrubbing.
- **Hierarchies:** drop a parent null, all children inherit visibility via `MODE_UNDEF` (C4D default). Standard behavior.
- **Overlap on same object:** visible if ANY clip covers the frame (union across all clips and tracks). Same rule as multi-track audio.
- **Outside all object clips for that object:** object is hidden. Timeline owns the object's visibility entirely while any clip exists for it.
- **Pre-existing visibility animation:** take over and warn. Drop creates the clip; toast/banner tells the user "this object's visibility animation has been overridden by Shotblocks; remove existing track if you want to revert." Less destructive than refusing.
- **Visual style:** new clip color + icon (Mike provides Figma at implementation). Read as "still a clip, different category."
- **Clip span semantics:** same as camera clips. Trim/roll edges, drag body, razor splits, all standard NLE gestures apply unchanged.
- **No orphan distinction needed yet:** if the underlying object is deleted, treat as orphan (same visual pattern as orphan camera clips — already shipping). Reuse the existing orphan plumbing.
